Understanding Server Components in Next.js 16

MAR 14, 2026 DEV 2.4K CHARS BY JIWON

React Server Components represent a fundamental shift in how we think about building React applications. Instead of shipping JavaScript for every component to the browser, server components render entirely on the server and send only the HTML result.

What Are Server Components?

At their core, Server Components are React components that execute exclusively on the server. They can directly access databases, file systems, and other server-side resources without exposing those operations to the client.

Server Components let you write components that render on the server and have zero impact on your client bundle size.

The Mental Model

Think of it this way: your component tree is split into two worlds. The server world can do anything a Node.js process can do. The client world is what runs in the browser. The boundary between them is the 'use client' directive.

// This is a Server Component (default)
async function PostList() {
  const posts = await db.posts.findMany()
  return (
    <ul>
      {posts.map(post => (
        <li key={post.id}>{post.title}</li>
      ))}
    </ul>
  )
}

When to Use Client Components

You need client components when your UI requires:

  • Event handlers (onClick, onChange)
  • Browser APIs (window, localStorage)
  • React hooks (useState, useEffect)
  • Third-party libraries that use browser APIs

Common Pitfalls

The most common mistake is marking too many components as client components. Start server-first and only add 'use client' when you need interactivity. Push client boundaries as low in the tree as possible.

Another pitfall: passing non-serializable props across the server-client boundary. Functions, class instances, and Symbols cannot cross this boundary. Stick to plain objects, arrays, and primitives.

Conclusion

Server Components are not just a performance optimization — they represent a new architecture for React applications. By default, your code runs on the server. You opt into the client only when you need it. This mental model leads to faster, lighter applications with better developer ergonomics.
